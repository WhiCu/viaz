# viaz — multipath connection bonding для обхода блокировок

Библиотека на Go, объединяющая несколько физических соединений (путей) в один
надёжный логический поток (`net.Conn`). В отличие от smux/yamux (мультиплексация
N логических потоков над ОДНИМ физическим соединением), `viaz` решает обратную
задачу: ОДИН логический поток размазывается по N физическим соединениям —
для пропускной способности, отказоустойчивости и устойчивости к ошибкам
отдельных каналов.

## Зачем

- Существующие реализации (`getlantern/multipath`) экспериментальны: нет SACK, примитивная схема ретрансмиссии, статичный незашифрованный handshake-заголовок.
- Цель проекта — рабочий, тестируемый аналог: пути транспортируются через KCP + smux, протокол спроектирован так, чтобы слой обфускации можно было подключить отдельно, не трогая ядро.

## Структура репозитория

| Пакет                 | Отвечает за                                              |
|-----------------------|-----------------------------------------------------------|
| `frame/`              | Кодек фреймов: encode/decode, varint, SACK delta-кодинг   |
| `mpconn/`             | Ядро: reorder-буфер, sequencing, flow control, сборка     |
| `pathstate/`          | RTT/RTO (Jacobson/Karels), AIMD cwnd, quarantine          |
| `scheduler/`          | Стратегии выбора пути (RoundRobin, WeightedRTT, Jitter)   |
| `subflow/`            | Интерфейс `Subflow` + мок-реализации для тестов           |
| `transport/kcp/`      | Адаптер KCP-сессии под интерфейс `Subflow`                 |
| `transport/smux/`     | Адаптер smux-stream поверх KCP-сессии под `Subflow`        |
| `asmcore/`            | ASM/C hot-path через `.syso` (см. GcToolchainTricks)       |

## Wire-протокол

### Общий заголовок фрейма

Первый байт — bit-packed:

| Биты   | Назначение                                  |
|--------|----------------------------------------------|
| 7–4    | Тип фрейма (см. таблицу ниже)                |
| 3      | Flag: ACK piggyback присутствует в этом фрейме |
| 2      | Flag: FIN (последний чанк потока)             |
| 1, 0   | Резерв                                        |

После первого байта — `streamID` (varint, QUIC-style: 2 старших бита первого
байта варинта определяют длину 1/2/4/8 байт). Для одного логического потока
на сессию `streamID = 0` → 1 байт.

### Типы фреймов

| Тип | Код | Статус       | Поля после общего заголовка                                                                 |
|-----|-----|--------------|-----------------------------------------------------------------------------------------------|
| DATA          | 1 | **зафиксировано** | `seq` (varint, frame-index, назначается на mpConn, не на subflow) · `length` (varint, длина payload в байтах) · `payload` (length байт) · *(опционально, если flag piggyback)* SACK-секция, формат как в ACK ниже |
| ACK           | 2 | **зафиксировано** | `cumAck` (varint, frame-index, все чанки до него подряд подтверждены) · `rangeCount` (varint) · далее rangeCount раз: `gap` (varint, сколько чанков пропущено перед диапазоном) + `len` (varint, длина подтверждённого диапазона) |
| PING          | 3 | проект (не зафиксировано) | предложение: `pingID` (varint) — для сопоставления с PONG |
| PONG          | 4 | проект (не зафиксировано) | предложение: `pingID` (varint), эхо из соответствующего PING |
| PATH_ADD      | 5 | проект (не зафиксировано) | предложение: `pathID` (varint) |
| PATH_REMOVE   | 6 | проект (не зафиксировано) | предложение: `pathID` (varint) + `reason` (1 байт код) |
| CLOSE         | 7 | проект (не зафиксировано) | предложение: `streamID` уже есть в общем заголовке; доп. `reason` (1 байт код) |
| PAD           | 8 | проект (не зафиксировано) | `length` (varint) + мусорный payload, игнорируется получателем |

**Зафиксировано и готово к реализации в Фазе 1: DATA и ACK** — именно на них
стоит писать кодек и тесты первым делом, остальные типы можно завести как
заглушки (`type` распознаётся, но кодек тела — `TODO`) и доделать по мере
подхода к ним в плане.

`seq` в DATA — это **порядковый номер чанка**, не байтовый offset (в отличие
от TCP). Растёт на 1 на каждый отправленный чанк логического потока,
независимо от того, через какой путь чанк в итоге уйдёт. Это и позволяет
получателю восстанавливать порядок при поступлении чанков с разных путей
вперемешку.

### Что отдельно от ядра протокола (не путать уровни)

- CID-handshake нового сабфлоу — отдельный механизм до того, как пошли фреймы
  (по аналогии с `getlantern/multipath`, но CID должен идти не в открытую, а
  под обфускацией — это задача транспортного/obfs-слоя, не `frame`-пакета).
- Обфускация, padding-таймеры, выбор размеров PAD — задача отдельного
  obfs-модуля над `Subflow`, ядро (`frame`/`mpconn`) ничего про это не знает.

## Тестирование

Каждая фаза включает свой набор тестов как обязательную часть, не опциональную:
table-driven unit-тесты, round-trip кодеки, fuzz-тесты декодера, race-тесты на
конкурентный доступ, хаос-тесты (искусственная деградация путей) и
differential-тесты для ASM hot-path против эталонной Go-реализации.
